## What I meant by the `payload.auth()` split point

When I said I found another subtle split point, I meant this: after we split the system into two auth collections, `admins` and `customers`, some server-rendered pages were still doing a very general auth check. In Payload, `payload.auth({ headers })` asks, "is there any authenticated user attached to this request?" It does not automatically mean "authenticated customer." It can also mean "authenticated admin," depending on which auth cookie is present. That matters because your storefront account pages, like account details and order history, are supposed to represent the customer side of the site. If a server page just checks `payload.auth()` and then assumes any authenticated person is a customer, an admin session could be mistaken for a storefront customer session. That is what I meant by an admin accidentally masquerading as a customer. The fix was not about login breaking. The fix was about making the account pages intentionally say, "only treat this request as valid if the authenticated collection is `customers`." That is why I added the dedicated helper in [getAuthenticatedCustomer.ts](/C:/Users/zhang/00My%20Stuff/Coding/Learning/learning-payload-cms/baked-with-blessings/src/utilities/getAuthenticatedCustomer.ts).

## What I meant by `the first patch bundle was too wide`

That comment was about implementation strategy, not about a design problem in your app. A "patch bundle" just means a batch of edits being applied together. This refactor touched multiple subsystems at once: collection definitions, access utilities, storefront auth, admin auth, server-rendered account pages, seed data, tests, generated types, and bootstrap behavior. When a change set is that wide, it becomes harder to tell whether a failure comes from the underlying design or just from one file having changed since the plan was formed. When I said the first patch bundle was too wide, I meant that trying to land the entire refactor in one giant edit pass was making it harder to safely validate. So I split the work into smaller slices, such as collections first, then auth paths, then account pages, then tests. That is a normal engineering tactic to reduce accidental breakage while still moving toward the same final architecture.

## What I meant by `reduce schema churn`

"Schema churn" means unnecessary database-structure changes while a refactor is happening. A database schema is the shape of your database: tables, columns, constraints, foreign keys, and how other tables point at one another. Churn means lots of renames, new tables, remapped references, or migration prompts happening all at once. Reducing schema churn does not mean "avoid refactoring because refactoring is scary." It means "do not create extra database changes unless they buy you something important." In this case, the ecommerce side of your app already had customer-related data and relationships built around what used to be the `users` collection. By keeping the new `customers` collection mapped to the old `users` table, I was trying to preserve those existing customer-side relationships while still separating admin auth into its own `admins` table. That was a practical move, not a philosophical one. It was meant to keep the auth split focused on the auth split instead of also forcing a full customer-table migration at the same moment.

## What the database design is right now

Right now, the implementation is not "one `users` table with both admins and customers mixed together." It is closer to this: admins live in a real `admins` collection and a real `admins` database table, while customers live in a `customers` collection at the Payload application level but that collection is still physically backed by the old `users` database table because of `dbName: 'users'` in [Customers/index.ts](/C:/Users/zhang/00My%20Stuff/Coding/Learning/learning-payload-cms/baked-with-blessings/src/collections/Customers/index.ts). So there are two auth collections in the app. There are also two physical auth tables involved at the database level: `admins` for admins, and `users` for customers. The name `users` is now acting as a legacy physical table name for customer records, not as the original shared `users with roles` model.

## Are admins and customers in one table right now

No. Admins are not being stored in that old `users` table anymore. Admins are stored in `admins`. Customers are stored in the old `users` table, but through the new `customers` collection definition. So the current implementation is already a real split in the important security sense: admin auth is isolated from customer auth. What is still "legacy" is the physical table name on the customer side. That is the nuance. The logical model is split. The physical customer table name is not yet renamed to `customers`.

## Why your hunch that something felt off was reasonable

Your hunch was reasonable because `slug: 'customers'` combined with `dbName: 'users'` can absolutely feel strange at first glance. It means the app code speaks in the new language, `customers`, while the database still uses the older physical name, `users`, for that side of the system. That is a valid transitional or compatibility-oriented design, but it is not the cleanest possible design if you are optimizing for conceptual neatness. So if your instinct was, "wait, are we only half-splitting this?", that instinct was fair. The answer is that we fully split admin auth from customer auth, but we did not fully rename the customer table at the database layer.

## Could `customers` turn out to be a bad name later

It is possible, but in your app it is not the most likely failure mode. `Customers` is a very good name if this collection represents people who buy products, have carts, place orders, store addresses, and use the storefront account area. That is exactly what this repo is doing today. In fact, your ecommerce plugin wiring already speaks that language directly in [plugins/index.ts](/C:/Users/zhang/00My%20Stuff/Coding/Learning/learning-payload-cms/baked-with-blessings/src/plugins/index.ts), where orders, carts, addresses, and storefront auth all revolve around a customer concept. So for the current domain, `customers` is not some awkward compromise. It is a strong domain name.

The name starts becoming questionable only if you later decide this collection is supposed to represent a broader public identity system, not just buyers. For example, if you later want one shared public collection for newsletter-only members, event attendees, wholesale leads, loyalty members who have not purchased yet, or some broader "community account" concept, then `accounts`, `members`, or `people` could be more future-proof. But that is a different product direction than the one your repo expresses today. Given the app you are actually building now, I think regretting `customers` later is fairly unlikely. Regretting a half-clean split is more likely than regretting the word `customers`.

## What moving customer-side data and relationships would really mean here

In this repo, moving the customer side off the old `users` table would not just mean renaming one file. It would mean changing both the physical storage location and the relationships that depend on it. Right now, the app-level collection name is already `customers`, and your storefront code already calls `/api/customers`, your provider already logs into `customers`, and the ecommerce plugin already says the customer collection slug is `customers`. So the app layer is largely done. The remaining work is at the database layer.

Concretely, the first step would be to stop using `dbName: 'users'` in [Customers/index.ts](/C:/Users/zhang/00My%20Stuff/Coding/Learning/learning-payload-cms/baked-with-blessings/src/collections/Customers/index.ts). Once you do that, Payload will want a real `customers` table instead of continuing to use the legacy `users` table. Then you have to make sure the customer records themselves move over. If you already have customer rows in `users`, those rows need to exist in `customers` instead. In an early-stage project with little or no real data, that can be as simple as recreating or reseeding them. If there is meaningful data, then you would care about copying the records while preserving their IDs so connected data still points to the same person.

The next step is the relationships. In your repo, the ecommerce side already has several places where other records point at a customer. The most obvious ones are in the seed flow and plugin behavior: `addresses` store a `customer`, `transactions` store a `customer`, `carts` store a `customer`, and `orders` store a `customer`, all visible in [src/endpoints/seed/index.ts](/C:/Users/zhang/00My%20Stuff/Coding/Learning/learning-payload-cms/baked-with-blessings/src/endpoints/seed/index.ts). On top of that, the `customers` collection defines join fields for `orders`, `cart`, and `addresses`, all through `on: 'customer'` in [Customers/index.ts](/C:/Users/zhang/00My%20Stuff/Coding/Learning/learning-payload-cms/baked-with-blessings/src/collections/Customers/index.ts). At the app level, those relationships already say "customer." The part that may still be legacy is the underlying database constraint or foreign-key target if it still points at the old `users` table.

So when I wrote "updating whatever foreign keys or join references still point at that table," I meant this: after creating a real `customers` table, the database has to understand that `orders.customer`, `carts.customer`, `transactions.customer`, and `addresses.customer` should point at `customers.id`, not `users.id`. If the customer IDs are preserved during migration, the values in those relationship columns may not need to change, but the database constraints and generated schema metadata still do. If the IDs are not preserved, then both the target table and the stored relationship values would need updating, which is messier.

There is also a broader Payload-system angle. Auth collections often carry auth-related storage concerns, such as login fields, reset-password fields, and sometimes supporting tables depending on the auth configuration. So finishing the migration cleanly is not just "move buyer rows." It is "make the customer auth collection fully and physically be `customers` everywhere that matters." That is why I described it as a deliberate schema migration rather than a tiny rename.

## What I would recommend next if you want a cleaner early-stage schema

Because you are still early, I think there is a good argument for one more cleanup step: make the customer side physically use a real `customers` database table instead of reusing the old `users` table name. That would make the system easier to reason about because the logical names and physical names would match. You would then have `admins` in `admins` and `customers` in `customers`, which is cleaner and less surprising. The tradeoff is that this is a true schema migration. It means moving the customer-side data and relationships off the old `users` table, updating whatever foreign keys or join references still point at that table, and reconciling the database state accordingly. In other words, your concern is not wrong. The current design is secure enough for the auth split goal, but it is not the cleanest final schema if you want the database itself to read as cleanly as the app code. If you want, the next step can be a deliberate follow-up refactor whose only job is: "finish the split all the way down to the database table names so customers no longer live in the legacy `users` table."
